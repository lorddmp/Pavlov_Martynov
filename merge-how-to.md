#How to use merge

В Git для получения ветки с сервера и слияния одной ветки с другой используются разные команды, а после слияния могут потребоваться определённые действия. 

Получение ветки с сервера
Команда git pull забирает изменения с удалённого репозитория и пытается слить их с текущей локальной веткой. По сути, это комбинация двух других команд: git fetch (скачивает изменения, но не применяет их) и git merge (сливает скачанные изменения с веткой). 

Общая форма команды: git pull <remote> <branch>. Например: git pull origin main — получить изменения из ветки main на удалённом репозитории origin и слить их в текущую локальную ветку. 

Важно: git pull всегда обновляет только текущую ветку, а не все ветки сразу. Если не указывать ни remote, ни ветку, Git использует настройки текущей ветки: какой remote она отслеживает и какую удалённую ветку с ней связывает. 

Слияние веток
Команда git merge сливает изменения из одной ветки в другую. Обычно нужно переключиться на ветку, в которую нужно влить изменения (например, main или develop), а затем выполнить git merge с указанием ветки, из которой нужно забрать изменения (например, feature-ветка). 

Если конфликтов нет, Git создаст новый «коммит слияния» (merge commit), у которого будет два родительских коммита. Если есть конфликты, Git остановит процесс, отметит конфликты в файлах, и нужно будет разрешить их вручную, добавить исправленные файлы (git add) и завершить слияние командой git commit. 

Некоторые опции команды git merge:
--ff — использовать fast-forward, если возможно (по умолчанию);
--no-ff — создать коммит слияния даже при возможности fast-forward;
--ff-only — выполнить слияние, только если возможен fast-forward;
--squash — объединить все коммиты вливаемой ветки в один.

Действия после слияния
Если возникли конфликты, необходимо их разрешить и завершить слияние вручную. Процесс разрешения конфликтов: 

Git сообщает о конфликте при выполнении git merge или git rebase.
Git помечает конфликтующие файлы специальными маркерами.
Пользователь редактирует файлы, разрешая конфликты.
Исправленные файлы добавляются в индекс: git add <resolved-file>.
 
После слияния веток одна из них становится лишней — её можно удалить командой git branch -d. 

Важно: успешное разрешение конфликтов требует не только технических навыков, но и понимания контекста изменений. При сомнениях всегда рекомендуется консультироваться с автором конфликтующих изменений.
